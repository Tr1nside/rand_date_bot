# 💘 Date Randomizer Bot — DESIGN.md

## Концепция

Телеграм-бот, который предлагает случайное свидание под параметры пользователя.
Пользователь указывает фильтры → получает рандомное свидание → лайкает / отмечает как посещённое.

## Модели БД

### `dates` — свидания

| Поле | Тип | Описание |
| ------ | ----- | ---------- |
| `id` | INTEGER PK | — |
| `description` | TEXT | Описание свидания |
| `cash` | INTEGER (1–3) | Уровень затрат |
| `time` | INTEGER | Длительность в часах |
| `is_home` | BOOLEAN | `True` = дома, `False` = вне дома |
| `photo_file_id` | TEXT | Telegram file_id фото |

### `users` — пользователи

| Поле | Тип | Описание |
| --- | --- | --- |
| `id` | INTEGER PK | Telegram user_id |
| `username` | TEXT | — |
| `registered_at` | DATETIME | — |
| `is_admin` | BOOLEAN | Права админа |

### `user_history` — история взаимодействий

| Поле | Тип | Описание |
| --- | --- | --- |
| `user_id` | INTEGER FK | — |
| `date_id` | INTEGER FK | — |
| `is_liked` | BOOLEAN | `True` = лайк |
| `dropped_at` | DATETIME \| NULL | Когда отметил «сходили» |

**Уникальный индекс:** `UNIQUE(user_id, date_id)` — одна запись на пару.

## Логика рандома

**Запрос:**

```sql
SELECT * FROM dates
WHERE cash <= :max_cash
  AND time <= :max_time
  AND is_home = :is_home
  AND id NOT IN (
    SELECT date_id FROM user_history
    WHERE user_id = :user_id AND dropped_at IS NOT NULL
  )
ORDER BY RANDOM()
LIMIT 1
```

- Посещённые (`dropped_at IS NOT NULL`) исключаются **навсегда**
- Лайкнутые могут попасться снова — это нормально, данные нужны для будущих рекомендаций
- Если результат пуст → сообщение «Ничего не найдено 😔»

## UX-флоу пользователя

```bash
/start
  └── Приветствие + кнопка «Найти свидание»

«Найти свидание»
  └── Где проведём? [🏠 Дома] [🌆 Вне дома]
        └── Бюджет? [💸 1] [💸💸 2] [💸💸💸 3]
              └── Сколько времени? [1ч] [2ч] [3ч] [4ч+]
                    └── Карточка свидания:
                          [фото]
                          Описание...
                          ────────────────
                          [❤️ Лайк]  [✅ Сходили]  [🔄 Другое]
```

**Кнопки на карточке:**

- `❤️ Лайк` — ставит/снимает лайк (тоггл), запись в `user_history.is_liked`
- `✅ Сходили` — пишет `dropped_at = now()`, свидание больше не появится
- `🔄 Другое` — выдать другое свидание с теми же фильтрами (без сброса фильтров)

На каждом шаге доступны кнопки: `[← Назад]` `[✖ Отмена]`

### Управление админами

```bash
/add_admin  → ввести Telegram ID нового админа
/remove_admin → ввести Telegram ID для удаления
/list_admins  → список всех админов
```

Добавлять/удалять админов может **любой** действующий админ.

## Middleware

`AdminCheckMiddleware` — перед каждым апдейтом для admin-хендлеров проверяет `users.is_admin` в БД. Если не админ — игнорирует или отвечает «Нет доступа».

## Будущие фичи (не в MVP)

- **История свиданий** — список посещённых (`dropped_at IS NOT NULL`)
- **Рекомендации** — алгоритм k ближайших соседей на основе `is_liked`
- **Статистика для админа** — самые лайкаемые свидания, активность

---

## Что НЕ входит в MVP

- Миграции (alembic) — `Base.metadata.create_all()` достаточно для старта
- Docker / деплой
- Webhook (используем polling)
- Пользовательские предпочтения (фильтры вводятся каждый раз)
